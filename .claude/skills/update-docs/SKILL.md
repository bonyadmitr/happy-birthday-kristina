# Синхронизация документации с кодом

Правило простое: **меняешь поведение или структуру сайта → обновляешь доку в том же изменении, до коммита.** Доки маленькие и держатся в актуальном состоянии осознанно.

## Когда применять

Любая правка, меняющая **как сайт работает или из чего состоит**:
- разметка (`index.html`), стили/палитра/анимации (`css/styles.css`), логика (`js/main.js`);
- новые файлы/секции/ассеты, переименования, удаление;
- поведенческие фичи в `js/config.js` (не сам текст поздравления, а структура/новые поля).

**Не нужно** для: правки текстов/имени/значений в `js/config.js` (персонализация), изменений только в `tests/`, `playwright.config.js`, `package.json`.

Сомневаешься — лучше сверься с доком: это дёшево.

## Что делать

**Перед правкой** — открой релевантный раздел и учти «Гочи»:
- [CLAUDE.md](../../../CLAUDE.md) — карта проекта, ключевые файлы, **гочи (iOS, background-clip, оптический центр «Л», зум, overscroll, floaties)**.
- [docs/DOCUMENTATION.md](../../../docs/DOCUMENTATION.md) — технический разбор по секциям.

**После правки** — синхронизируй:

| Что изменил | Что обновить |
|---|---|
| Поток сцен, интерактив (распаковка, «Л», кнопки) | `DOCUMENTATION.md` §2 + при необходимости `CLAUDE.md` «Сцена» |
| Фон/палитра/шрифты/атмосфера (`:root`, `.bg`) | `DOCUMENTATION.md` §3 |
| Новый/изменённый паттерн, из-за которого легко ошибиться (iOS-специфика и т.п.) | добавь **гочу** в `CLAUDE.md` |
| Частицы, музыка, reveal, зум, safe-area | соответствующий раздел `DOCUMENTATION.md` (§5–§7) |
| Буква «Л»: шрифт/размер/клиппинг/центр | `DOCUMENTATION.md` §8 (перепровер оптический центр по сниппету там же) |
| Тесты, запуск, деплой, структура файлов | `DOCUMENTATION.md` §9–§10, `README.md` |
| Новый файл/папка | дерево структуры в `DOCUMENTATION.md` §1 и в `README.md` |
| Персонализация (новое поле config) | секция «Персонализация» в `CLAUDE.md`/`README.md` |

## Как писать

- **Не выдумывай.** Имена файлов/функций/значения бери из кода (быстрый `grep`/`Read`), не по памяти.
- Пиши коротко и по факту — доки этого проекта намеренно компактные.
- Если код противоречит доку — приоритет у кода: сначала подтверди, что изменение намеренное, потом обнови док.

## Проверка

После правок кода+доков прогони `npm test` (см. `DOCUMENTATION.md` §9) и убедись, что упомянутые в доке факты (пути, значения) соответствуют реальности.
